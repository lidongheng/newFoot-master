## 工作流程总览

对于一场比赛赛后分析，每场比赛有不同的情况，需要从基础数据中挖掘有用的信息，以了解比赛的过程。

整个流程分为以下步骤：

1. **查阅和分析本场比赛的大名单、首发及替补阵容、赛后统计数据，挖掘有用的信息**
2. **查阅和分析本场比赛的亚盘指数，确定哪个队赢下了指数**
3. **查阅和分析本赛季联赛、主队在主场或客队在客场打同档次球队或同初赔球队的输赢**
4. **取近6轮、近12轮主队和客队的联赛比赛结果和积分榜进行分析。涉及到球队换帅等其他因素时，还可以指定任意轮次**
5. **查阅比赛事件，分析比赛走向和事件发生后推测球队可能的安排**

## 第一步：查阅和分析本场比赛的大名单、首发及替补阵容、赛后统计数据，挖掘有用的信息

### 目标
获取主队和客队的完整大名单信息，包括每个球员的出场次数、位置、进球助攻数据等。
根据大名单和首发阵型排出首发阵容，加入到赛后分析报告中
对赛后统计数据进行对比，找到赢球球队赢球或输球球队输球或平局的原因

大名单已经由用户保存在`league-analyzer/news/${联赛名}/${赛季}/${轮次}/${对阵双方}`
首发及替补阵容、比赛事件、赛后统计数据保存在`backend-server/basicData/${联赛名}/${赛季}/${轮次}/${对阵双方}`

### 执行步骤

#### 1.1 读取大名单

从 `league-analyzer/news/{leagueSlug}/{season}/round-{轮次}/{对阵双方}/` 中读取主队和客队的大名单文件。

大名单格式示例（每行一位球员）：
```
号码-姓名 出场数场首发数首发 进球数球助攻数助 {位置JSON} 国籍 年龄 身高 身价
```

关注以下信息：
- **主力球员**：首发率高（>70%）的球员是否全部出场
- **伤停球员**：缺阵球员对球队实力的影响
- **轮换情况**：本场首发与常规首发的差异，推断教练的战术意图
- **年轻球员**：是否有年轻球员获得机会，可能影响比赛稳定性

#### 1.2 读取首发阵容

从 `lineup.json` 中获取双方阵型和首发名单。

关键字段：
- `homeFormation` / `guestFormation` — 阵型（如 `"4-2-3-1"`）
- `homeStarting` / `guestStarting` — 首发球员列表
- `homeSub` / `guestSub` — 替补球员列表

将首发阵容按以下格式整理输出（结合大名单中的球员详细信息）：

#### 首发阵容格式规范 ⭐⭐⭐

**格式说明**:
- 位置排列：从右向左（门将/后卫线/中场线/前场线/前锋）
- 分隔符：用 `/` 分隔不同位置线，用 `,` 分隔同一位置线的球员
- 替补说明：在括号内标注位置
- **必须包含四个部分**：预测首发（11人）、替补（9人）、伤停、落选

**完整格式示例 - 曼城 (4-1-4-1)**:
```
预测首发：25-多纳鲁马/27-努内斯，3-迪亚斯，24-格瓦迪奥尔，33-尼科奥赖利/14-尼古拉斯.冈萨雷斯/11-多库，20-贝尔纳多·席尔瓦，4-赖因德斯，47-福登/9-哈兰德

替补：1-特拉福德(GK)，82-里科·刘易斯(RB)，5-斯通斯(CB)，45-胡桑诺夫(CB)，21-艾特努里(LB)，26-萨维尼奥(LW)，7-马尔穆什(LW)，10-切尔基(CAM)，52-奥斯卡·鲍伯(RW)

伤停：
- 16-罗德里·赫尔南德斯（29岁，191cm，西班牙，9000万，6场4首发）- 腿筋伤
- 8-马特奥·科瓦契奇（31岁，177cm，克罗地亚，2000万，1场0首发）
- 44-菲利普斯（30岁，179cm，英格兰，1000万）

落选：
- 6-内森·阿克（30岁，180cm，荷兰，2200万，5场1首发）
```

#### 1.3 分析赛后技术统计

从 `techStats.json` 中读取 `fullMatch` 字段，对比双方关键指标。

**核心指标分析优先级**：

| 优先级 | 指标 | 键名 | kind | 分析要点 |
|--------|------|------|------|----------|
| ⭐⭐⭐ | 控球率 | `控球率` | `CONTROL_PERCENT` | 哪一方主导比赛节奏 |
| ⭐⭐⭐ | 射门/射正 | `射门`/`射正` | `SHOOT`/`TARGET` | 进攻效率，射正率=射正/射门 |
| ⭐⭐⭐ | 预期进球(xG) | `预期进球(xG)` 或 `预期进球` | `52` | 射门质量，xG与实际进球的差异 |
| ⭐⭐⭐ | 得分机会（绝佳机会） | `得分机会` | `57` | 创造了多少绝佳得分机会 |
| ⭐⭐⭐ | 错失得分机会 | `错失得分机会` | `58` | 浪费了多少绝佳机会，结合得分机会看把握能力 |
| ⭐⭐⭐ | 救球（扑救） | `救球` | `SAVE` | 门将扑救次数，反映门将发挥和对手射门威胁 |
| ⭐⭐ | 危险进攻 | `危险进攻` | `DANGEROUS_ATTACK` | 深入对方腹地的进攻次数 |
| ⭐⭐ | 禁区内射球 | `禁区内射球` | `50` | 高质量射门占比，禁区内比禁区外威胁大得多 |
| ⭐⭐ | 对方禁区触球 | `对方禁区触球` | `59` | 在对方禁区的活跃程度 |
| ⭐⭐ | 传球/传球成功率 | `传球`/`传球成功率` | `PASSBALL`/`PASSBALL_SUCCESS_PERCENT` | 传控能力 |
| ⭐⭐ | 成功传中 | `成功传中` | `SUCCESS_CROSS` | 边路进攻效果 |
| ⭐⭐ | 角球 | `角球` | `CORNER` | 进攻压迫力和定位球机会 |
| ⭐⭐ | 犯规/黄牌 | `犯规`/`黄牌` | `FOULS`/`YELLOW` | 比赛烈度和纪律性 |
| ⭐⭐ | 成功抢断 | `成功抢断` | `STEAL_SUCCESS` | 防守积极性和中场争夺能力 |
| ⭐⭐ | 阻截 | `阻截` | `HOLDUP` | 防守站位和拦截意识 |
| ⭐⭐ | 解围 | `解围` | `60` | 被动防守次数，多=承受压力大 |
| ⭐ | 运动战xG/定位球xG | `运动战预期进球`/`定位球预期进球` | `53`/`54` | xG来源拆分：运动战还是定位球威胁大 |
| ⭐ | 射正预期进球 | `射正预期进球` | `56` | 射正的含金量（PSxG） |
| ⭐ | 过人 | `过人` | `DRIBBLES` | 个人突破能力 |
| ⭐ | 1对1成功 | `1对1成功` | `64` | 对抗成功率 |
| ⭐ | 争顶成功 | `争顶成功` | `63` | 制空权 |
| ⭐ | 长传 | `长传` | `LONG_PASS` | 长传转移球风格 |
| ⭐ | 铲球 | `铲球` | `TACKLE` | 防守凶悍程度 |

**分析逻辑**：
- **控球率差距 >15%**：控球方通常掌握比赛节奏，但需结合射门数判断是否有效控球
- **射门多但射正少**：进攻缺乏精度，远射多或进攻组织差
- **xG > 实际进球**：运气不佳或临门一脚差；**xG < 实际进球**：效率极高或对手门将发挥差
- **得分机会多但错失也多**：把握能力差，临门一脚有问题
- **救球多**：门将高光表现，可能掩盖了防守端的问题；也可能对手射门多但被门将化解
- **禁区内射球占比高**：进攻质量好，射门都在禁区内完成
- **角球数多**：持续围攻对方半场，定位球威胁大
- **定位球xG > 运动战xG**：球队依赖定位球得分，运动战创造力不足
- **犯规多+黄牌多**：比赛节奏被打断多，可能是弱队的防守策略
- **解围多**：防守端压力大，长期被对手围攻

#### 1.4 分析球员统计

从 `playerStats.json` 中提取关键球员的表现数据。

关键字段（`stats` 对象中）：
- `Shots` / `ShotOnTarget` — 射门/射正
- `KeyPass` — 关键传球
- `PassSucceeRatio` — 传球成功率
- `BodyContact` — 身体对抗
- `Steal` / `Intercept` — 抢断/拦截
- `Ranking` — 球员评分
- `ControlPass` — 控球
- `Clearances` — 解围

重点关注：
- **评分最高/最低的球员** — 谁是本场最佳，谁拖了后腿
- **射门王** — 哪些球员创造了最多威胁
- **关键传球王** — 哪些球员是进攻组织核心
- **抢断/拦截最多** — 哪些球员防守贡献大

## 第二步：查阅和分析本场比赛的亚盘指数，确定哪个队赢下了指数

### 目标
分析初盘让球数，判断博彩公司对比赛走向的预期，以及实际结果是否符合预期。

### 执行步骤

#### 2.1 获取本场比赛的盘口数据

从 `backend-server/match_center/s{leagueSerial}.js` 中找到目标比赛，读取数组中的关键字段：

| 索引 | 含义 | 示例 |
|------|------|------|
| 6 | 全场比分 | `'1-1'` |
| 7 | 半场比分 | `'0-0'` |
| 10 | 初盘全场让球 | `-0.75`（负数=主队受让） |
| 11 | 初盘半场让球 | `-0.25` |
| 12 | 初盘大小球 | `'2.5/3'` |

#### 2.2 解读让球盘

**让球含义**：
- 正数（如 `1.25`）：主队让球（主队实力强）
- 负数（如 `-0.75`）：主队受让（客队实力强，即客队让球）
- `0`：平手盘（双方实力接近）

**判断标准**：
- 计算 `主队进球 - 客队进球 - 让球数`：
  - 结果 > 0.25 → 主队赢盘
  - 结果 = 0.25 → 主队赢半
  - 结果 = 0 → 走水（退款）
  - 结果 = -0.25 → 主队输半
  - 结果 < -0.25 → 主队输盘

#### 2.3 结合球队分档分析

对照 `references/{leagueSlug}-team-strength.md` 中的球队分档和理论盘口：
- 实际初盘与理论盘口是否一致？差异说明什么？
- 比如理论上主让半一，实际开出主让一球——说明博彩公司更看好主队

#### 2.4 运行历史同赔 + 同档次对阵分析

```bash
cd backend-server
node crawlerHistoryHandicap.js
```

此脚本现在输出两部分内容：

**Part 1: 历史同赔走势**
- 主队在主场、同让球数的历史战绩和赢盘率
- 客队在客场、同让球数的历史战绩和赢盘率

**Part 2: 同档次对阵分析**（需要 `league-team-strength.json` 有对应联赛/赛季的分档数据）
- 主队在主场 vs 客队所在档次的所有球队：逐场列出轮次、对手、比分、让球盘口、胜负和盘路
- 客队在客场 vs 主队所在档次的所有球队：逐场列出轮次、对手、比分、让球盘口、胜负和盘路
- 每组统计摘要包含：胜平负、进失球、胜率、不败率、盘口赢盘率

**配置依赖**：
- `config/wudaconfig.js`：`leagueSerial`、`leagueSlug`、`season`、`roundSerial`、`matchSerial`
- `league-team-strength.json`：球队分档数据（格式：`{联赛slug: {赛季: {版本号: {档次: [球队名]}}}}`）

分析要点：
- 赢盘率 >60% 表示该队在同类盘口下表现稳定
- 赢盘率 <40% 表示该队经常被打穿
- 同档次对阵中，关注胜率和不败率——不败率高但胜率低说明平局多，打不穿盘口
- 对比同档次的比赛结果和盘口输赢——比赛不输但盘口常输，说明赢得不够多

## 第三步：查阅和分析本赛季联赛、主队在主场或客队在客场打同档次球队或同初赔球队的输赢

### 目标
分析主队在主场对阵同级别对手的战绩，以及客队在客场对阵同级别对手的战绩，评估双方在特定环境下的真实竞争力。

> **注意**：此步骤的数据已由 `crawlerHistoryHandicap.js` 的 Part 2 自动输出，无需额外操作。以下是分析方法说明。

### 数据来源

球队分档定义在两处（内容一致）：
- `references/{leagueSlug}-team-strength.md` — 人可读版本，含理论盘口
- `backend-server/league-team-strength.json` — 程序可读版本，脚本自动加载

示例（英超25-26赛季）：
- 第一档：阿森纳、曼城、切尔西、利物浦
- 第二档：热刺、曼联、阿斯顿维拉、纽卡斯尔联
- 第三档：埃弗顿、布莱顿、水晶宫、伯恩茅斯
- 第四档：诺丁汉森林、布伦特福德、富勒姆、桑德兰
- 第五档：西汉姆联、利兹联、伯恩利、狼队

### 筛选逻辑

`crawlerHistoryHandicap.js` 中的同档次筛选规则：

**主队主场打同档次**：
- `homeTeamId === 目标主队ID`
- `awayTeamId` 所在档次 === 目标客队所在档次
- 比赛已完赛且在目标比赛之前

**客队客场打同档次**：
- `awayTeamId === 目标客队ID`
- `homeTeamId` 所在档次 === 目标主队所在档次
- 比赛已完赛且在目标比赛之前

### 分析要点

脚本输出的同档次表格包含每场比赛的：轮次、日期、主队、比分、客队、让球、胜负（比赛结果）、盘路（盘口输赢）、半场比分。

从这些数据中分析：
- 主队主场对同档次强不强？——反映主场优势程度
- 客队客场对同档次稳不稳？——反映客场抗压能力
- 每场比赛的具体盘口不同（因为对手不完全相同），注意区分让球大小的差异
- 与全赛季总战绩对比，主客场是否有明显差异

## 第四步：取近6轮、近12轮主队和客队的联赛比赛结果和积分榜进行分析

### 目标
通过近期战绩判断双方当前状态，是上升期、平稳期还是下滑期。

### 执行步骤

#### 4.1 运行自定义积分榜脚本

使用 `crawlerCustomRank.js` 输出近N轮积分榜：

```bash
cd backend-server

# 近6轮积分榜（假设当前是第26轮）
node crawlerCustomRank.js --rounds 21-26

# 近12轮积分榜
node crawlerCustomRank.js --rounds 15-26

# 全赛季积分榜（参照）
node crawlerCustomRank.js --rounds 1-26
```

脚本输出包含：排名、球队、场次、胜平负、进失球、净胜球、积分、主客场战绩。

#### 4.2 分析近期趋势

从积分榜中提取分析要素：

| 指标 | 分析方向 |
|------|----------|
| 近6轮积分 vs 近12轮积分 | 近期是否加速/减速 |
| 近6轮排名 vs 全赛季排名 | 排名上升=状态好，下降=状态差 |
| 近6轮主场/客场分拆 | 是否主场龙客场虫，或反之 |
| 近6轮进球/失球 | 攻击力和防守稳定性的趋势 |
| 近6轮净胜球 | 正向大=碾压局多，负向大=被动局多 |

#### 4.3 特殊情况处理

涉及到球队换帅等其他因素时，可以指定任意轮次范围来单独观察：

```bash
# 比如某队第15轮换帅，看换帅后表现
node crawlerCustomRank.js --rounds 15-26

# 比如看赛季初磨合期
node crawlerCustomRank.js --rounds 1-6
```

#### 4.4 分析输出

结合三个维度（近6轮、近12轮、全赛季）做出判断：
- **上升期**：近6轮积分排名 > 近12轮 > 全赛季
- **平稳期**：三者排名接近
- **下滑期**：近6轮积分排名 < 近12轮 < 全赛季
- **波动期**：排名忽高忽低，不稳定

## 第五步：查阅比赛事件，分析比赛走向和事件发生后推测球队可能的安排

### 目标
通过比赛事件时间线还原比赛过程，分析关键事件（进球、红牌、换人等）对比赛走向的影响。

### 执行步骤

#### 5.1 读取比赛事件

从 `matchEvents.json` 中读取 `events` 数组。

事件类型（`type` 字段）：

| type | 含义 | 分析重要性 |
|------|------|-----------|
| `goal` | 进球 | ⭐⭐⭐ 改变比分 |
| `penaltyGoal` | 点球进球 | ⭐⭐⭐ 争议性/关键性 |
| `ownGoal` | 乌龙球 | ⭐⭐⭐ 意外事件 |
| `redCard` | 红牌 | ⭐⭐⭐ 少一人巨大影响 |
| `yellowCard` | 黄牌 | ⭐⭐ 纪律问题 |
| `substitution` | 换人 | ⭐⭐ 战术调整信号 |
| `secondYellow` | 两黄变红 | ⭐⭐⭐ 同红牌 |

每个事件的关键字段：
- `time` — 发生时间（如 `"30'"`、`"45+2'"`）
- `kind` — 主客队（`"HOME"` / `"AWAY"`）
- `player` — 涉及球员
- `assist` — 助攻球员（进球事件）
- `subOn` — 替换上场球员（换人事件）

#### 5.2 构建比赛时间线

按时间排序所有事件，划分为以下阶段：

| 阶段 | 时间段 | 关注点 |
|------|--------|--------|
| 开局阶段 | 0-15分钟 | 谁先取得控制权，有无早期黄牌/犯规 |
| 上半场中段 | 15-30分钟 | 阵型是否如预期运转 |
| 上半场尾段 | 30-45+分钟 | 半场前的关键事件 |
| 下半场开局 | 46-60分钟 | 中场休息后的战术调整，换人时间 |
| 下半场中段 | 60-75分钟 | 主要换人窗口，体能变化 |
| 下半场尾段 | 75-90+分钟 | 最后的进攻/防守调整 |

#### 5.3 分析关键转折点

对每个关键事件（进球、红牌、关键换人）分析：

1. **事件发生前的比赛状态**：比分、控球趋势、进攻压力
2. **事件本身**：谁进球/被罚下，进球方式（射门/头球/定位球）
3. **事件发生后的影响**：
   - 领先方是否收缩防守？
   - 落后方是否加大进攻？
   - 换人是进攻型还是防守型？
   - 是否改变了阵型？

#### 5.4 换人分析

换人是教练在场上唯一的战术调整手段，需要特别关注：

- **换人时机**：60分钟前换人通常说明战术不奏效或有伤病
- **换人类型**：
  - 同位置换人 → 体能管理
  - 增加前锋/攻击型中场 → 追分
  - 增加后卫/防守型中场 → 守住比分
- **三个换人窗口**：教练通常分3次使用5个换人名额，每次换人的时机和意图

#### 5.5 结合 matchInfo.json 的比赛摘要

`matchInfo.json` 中的 `postMatchSummary` 提供了AI生成的比赛摘要，可以与自己的分析进行交叉验证，但**不能直接复制作为分析结论**，需要用数据支撑或推翻其观点。
