# 比赛赛前导演推演工作流程

**SKILL**: dh-match-preview-director  
**创建时间**: 2026-06-21  
**最后更新时间**: 2026-06-21  
**版本**: v1.0

---

## 核心目标

把一场比赛先当成一部需要推演的电影，而不是一个单一赛果。

默认前提是用户已经先生成赛前分析报告，报告路径在 `cup-analyzer/${联赛}/report`。本技能先识别赛事模块，再读取对应赛前报告，再补充必要的本地数据或网络信息，最后写出 3 到 4 种可能剧情，并把产物保存到原比赛报告同级目录；赛中再用直播画面、数据节奏和盘口变化验证哪一条剧情正在发生。只有当比赛真实走势符合预设剧情，并且入场位置仍有价值时，才允许给出滚球投注建议。

投注建议必须服从用户指定的投注纪律文档；当前项目通用纪律参考为 `cup-analyzer/theWorldCup/每日投注纪律警醒.md` 和 `cup-analyzer/theWorldCup/投注组合拳策略.md`。

## 使用边界

适合：

1. 用户已经在 `cup-analyzer/${联赛}/report` 生成单场赛前分析报告。
2. 用户要在赛前报告基础上继续做导演式剧情推演。
3. 用户准备看直播并做滚球判断。
4. 用户希望得到 3 到 4 条比赛走势预案。

不适合：

1. 只要赛后复盘时，优先使用 `dh-post-match-analysis`。
2. 只要常规赛前报告时，优先使用 `dh-pre-match-analysis-report` 或联赛专属技能。
3. 赛前报告不存在时，先提示应生成赛前分析报告，或在用户明确要求时改用本地数据和网络搜索直接做观察预案。
4. 数据缺失到无法判断盘口和阵容时，先说明缺口，不编造信息。

## 四位专家角色

每次分析固定使用 4 个专家视角。专家可以独立陈述，但最终结论必须整合成一份统一的导演剧本。

### 基本面专家

负责判断比赛剧情是否有现实基础。

必须分析：

1. 用户 prompt 中补充的赛前新闻、发布会、赛前言论、战意类信息。
2. 报告里的伤停、伤疑、首发预测、轮换和关键球员状态。
3. 报告里的双方真实实力、球队档次和广实定位。
4. 报告里的战意、赛程、体能、心理压力和出线形势。
5. 报告里的主客场风格、控球倾向、反击能力。
6. 哪一方更可能主动，哪一方更可能等待。

### 基本面专家信息融合规则

基本面专家和盘口专家一样，必须同时接受用户 prompt 和赛前报告。

信息优先级：

1. 用户 prompt 中明确补充的最新赛前新闻、发布会、赛前言论、伤停伤疑，优先级最高。
2. 用户 prompt 没有基本面信息时，必须从 `cup-analyzer/${联赛}/report` 的赛前报告里提取全量基本面信息。
3. 赛前报告里的伤停、伤疑、首发预测、战意、赛程、球队状态、历史交锋、未来赛程都要作为剧情推演材料。
4. 用户 prompt 和报告冲突时，必须写清楚冲突点，并优先使用更新、更具体、来源更明确的信息。
5. 无法确认的新闻只能写成「待验证」，不能当作确定事实。

基本面专家要把信息转成剧情，而不是只罗列新闻。

转换方式：

1. 伤停或伤疑影响防线核心：推演防守不稳、被压制、容易丢定位球或后点。
2. 进攻核心复出或替补待命：推演下半场变招、后手进攻增强。
3. 发布会强调必须赢：推演主动开局、压上风险、后场空间。
4. 发布会强调先稳住或尊重对手：推演保守开局、半场小、僵局消耗。
5. 换帅、内部动员、关键战意：推演短期反弹、拼抢强度提升、受让方向更有韧性。
6. 赛程密集或未来关键战：推演轮换、节奏控制、领先后收缩。

基本面专家最终必须输出：

1. 最支持哪一种比赛剧情。
2. 哪些基本面信号会让该剧情成立。
3. 哪些基本面信号会让该剧情失效。
4. 需要在直播中重点观察的球员、位置和战术倾向。

### 盘口专家

负责判断盘口写出的隐含剧本。

必须分析：

1. 用户 prompt 中写明的赛前临场盘口，优先级高于报告中的早盘信息。
2. 初盘和临场让球是否符合广实定位。
3. 大小球是否支持开放比赛或谨慎比赛。
4. 让球深浅、水位舒适度、强弱双方拉力。
5. `_cycle.html` 趋势图里两队买入方向是否处于高位或低位。
6. 新闻变量是否支持盘口方向，例如换帅、核心复出、临场伤停、战意变化。
7. 赛中盘口变化是否和场面同频。
8. 哪些走势可能是强队虚热、弱队保护或诱导入场。

盘口专家必须先统一盘口方向：

1. 让球为正数时，通常代表主队让球。
2. 让球为负数时，通常代表客队让球。
3. 例如 `1.12 -1 0.78`，`-1` 代表客队让 1 球，左侧水位对应主队受让方向，右侧水位对应客队让球方向。

### 盘口专家稳定推演法

盘口专家的赛前推演要尽量稳定，不能只看单一盘口。

固定顺序：

1. 先读用户 prompt 中的临场亚盘和大小球。
2. 再读赛前报告里的初盘、广实定位、盘路榜和大小球榜。
3. 再读同场 `_cycle.html` 趋势图，判断哪一方持续走低，哪一方持续走高。
4. 再结合新闻变量，判断高低位背后是否有基本面解释。
5. 最后给出一条最稳定的盘口剧本，包含方向、比分想象和放弃条件。

示例：

用户写明突尼斯 vs 日本赛前盘口是亚盘 `1.12 -1 0.78`，大小球 `1.06 2.5 0.81`，其中 `-1` 代表客队日本让 1 球。盘口专家再看 `突尼斯_vs_日本_cycle.html` 趋势图，发现突尼斯一直走低，日本一直走高，突尼斯在低位，日本在高位。再结合新闻发现突尼斯换帅，存在「换帅如换刀」的反弹变量。此时盘口专家可以形成受让剧本：日本名气和让球拉力强，但市场趋势并不稳定支持日本穿盘，突尼斯低位受保护，推荐优先推演突尼斯 `+1`，比分想象为 `1-0` 或 `1-1`。

这类剧本必须写清楚：

1. 为什么受让方低位值得重视。
2. 为什么让球方高位不适合盲追。
3. 大小球 `2.5` 与比分想象是否一致。
4. 哪些直播信号证明受让剧本成立。
5. 哪些直播信号说明受让剧本失效。

### 解说员专家

负责在基本面专家和盘口专家分析完成后，拿到两者材料，再形成一个符合大众心理预期、容易被观众接受的赛前观点。

解说员专家的特点：

1. 必须先读取基本面专家和盘口专家的结论，不能独立抢先判断。
2. 必须发表明确观点，不能只做两边都可能的中性总结。
3. 观点可以在赛后被验证为正确或错误，但赛前必须符合大众心理预期。
4. 需要把复杂信息讲成观众听得懂、愿意相信的比赛故事。
5. 可以顺应热门球队、明星球员、换帅反弹、强队必须赢等大众叙事，但不能脱离基本面和盘口材料。

必须分析：

1. 基本面专家最支持的剧情。
2. 盘口专家最支持的剧情。
3. 大众最容易相信哪一种比赛故事。
4. 这个故事赛中最应该观察什么。
5. 如果直播走势符合大众预期，应该怎么解读。
6. 如果直播走势打脸大众预期，应该怎么切换剧情。

### 足球名宿专家

负责在解说员专家发表大众观点后，给出一个更辛辣、更有争议、与解说员形成对立的观点，用来吸引观众眼球。

足球名宿专家的特点：

1. 名宿已退役、淡出公众视线，要重新获得关注，必须有鲜明观点。
2. 观点要和解说员专家形成张力，最好能直接反驳解说员的主流叙事。
3. 可以辛辣、尖锐、反常识，但不能脱离现实，必须能从基本面、盘口或新闻中找到依据。
4. 不能为了哗众取宠而编造事实。
5. 名宿观点的价值是提示大众观点可能忽略的风险。

必须分析：

1. 解说员专家的观点哪里可能过于顺大众。
2. 强队是否只是名气强，真实杀伤未必足。
3. 弱队是否被低估，是否有守住或偷袭空间。
4. 教练和球员心理是否会让比赛反大众预期。
5. 比赛进入 60 分钟后，哪一方更可能先崩、先急或先冒险。
6. 哪个辛辣观点可以成为第三或第四剧情的核心。

## 标准工作流

### 第一步：读取赛前报告

本技能默认接在赛前分析报告之后使用。必须先识别 `${联赛}` 对应的模块目录，再从 `cup-analyzer/${联赛}/report` 定位并读取对应比赛报告。

常见 `${联赛}` 模块名：

1. 世界杯：`theWorldCup`
2. 英超：`epl`
3. 欧冠：`championsLeague`
4. 澳超：`aLeague`
5. 美职联：`mls`
6. 意甲：`serieA`
7. 韩K联：`koreanKLeague`

如果用户只给了对阵和轮次，没有给联赛，先根据当前上下文、报告目录和球队名称定位；仍无法唯一确定时，必须询问用户。

报告常见路径：

```text
cup-analyzer/${联赛}/report/{阶段}/round-{轮次}/{主队}_vs_{客队}.md
cup-analyzer/${联赛}/report/{阶段}/round-{轮次}/{主队}_vs_{客队}_cycle.html
cup-analyzer/${联赛}/report/{赛季}/round-{轮次}/{主队}_vs_{客队}.md
cup-analyzer/${联赛}/report/{赛季}/round-{轮次}/{主队}_vs_{客队}_cycle.html
```

读取报告后，先提取：

1. 主队、客队、阶段、轮次和比赛背景。
2. 赛前总判断和推荐方向。
3. 双方首发预测、伤停、阵容疑点。
4. 广实定位、初盘让球、大小球、半场盘。
5. 盘路榜、大小球榜、入球时间、历史同赔。
6. 战意、赛程、体能、出线形势。
7. 比赛 ID，即赛前报告第 7 行左右的 `比赛ID：` 字段。
8. 报告中已经写出的风险点和不确定点。

如果同场同时存在 `.md` 和 `_cycle.html`，优先读取 `.md`，再按需要查看 `_cycle.html` 中的周期图或补充结论。

如果用户 prompt 中写了临场盘口，必须单独记录，且不能被报告中的旧盘口覆盖。

如果用户 prompt 中写了赛前新闻、发布会、赛前言论、伤停或伤疑，必须单独记录，且不能被报告中的旧基本面信息覆盖。

### 第二步：补充本地数据和网络信息

读取赛前报告后，再按缺口补充信息：

1. 用户提供的信息。
2. 对应联赛或杯赛 `data` 下的赛程、初盘让球、半场盘、大小球。
3. 对应联赛或杯赛新闻、大名单、伤停、球队资料、历史同赔。
4. `cup-analyzer/${联赛}/workflow.md` 和同赛事规则。
5. 网络搜索中的最新伤停、首发消息、天气、临场新闻、赔率或盘口变化。

网络搜索规则：

1. 只有当赛前报告或本地数据缺少关键临场信息时才搜索。
2. 搜索到的信息必须标注来源和时间。
3. 网络信息与本地报告冲突时，必须明确写出冲突点，不直接覆盖原报告。
4. 无法确认的信息只能写成「待验证」，不能当作事实使用。

缺少关键数据时，必须明确写出「缺少什么」，并把结论降级为观察预案。

### 第三步：执行球队近期信息脚本

用户 prompt 和赛前报告里的信息通常不够，四位专家分析前必须使用脚本查询球队最近信息。

必须执行：

```bash
cd backend-server
node crawlerStatistics.js --matchSerial {比赛ID}
```

脚本用途：

1. 查询对赛往绩。
2. 查询主队近期战绩。
3. 查询客队近期战绩。
4. 查询主队未来比赛。
5. 查询客队未来比赛。
6. 查询主队伤病信息。
7. 查询客队伤病信息。

执行规则：

1. `{比赛ID}` 必须从对应赛前报告 `{主队}_vs_{客队}.md` 第 7 行左右的 `比赛ID：` 字段读取。
2. 不要为了运行该脚本修改 `backend-server/config/wudaconfig.js`。
3. `比赛ID：` 是赛前报告的必有字段，不需要从其他位置猜测或兜底查找。
4. 脚本输出必须进入四位专家的依据来源，不能只运行不使用。
5. 如果脚本因网络失败无法获取数据，要说明失败原因，并把最终产物降级为「缺少近期统计脚本校验的观察预案」。

四位专家必须使用脚本输出：

1. 基本面专家：用近况、未来赛程和伤病信息修正战意、体能和阵容判断。
2. 盘口专家：用对赛往绩、近期赢盘和大小球结果修正盘口剧本。
3. 解说员专家：用脚本输出包装成大众更容易接受的比赛故事。
4. 足球名宿专家：用脚本输出寻找反大众观点的现实依据。

### 第四步：专家按顺序给出判断

专家不是同时独立发言，必须按顺序推进：

1. 基本面专家先分析，给出基于阵容、伤停、战意、新闻和报告信息的剧情倾向。
2. 盘口专家再分析，给出基于临场盘口、初盘、大小球、`_cycle.html` 趋势图和新闻变量的盘口剧本。
3. 解说员专家必须拿到基本面专家和盘口专家的材料后，再输出一个符合大众心理预期的观点。
4. 足球名宿专家必须在解说员观点之后发言，给出一个辛辣、对立、但不脱离现实的观点。

每位专家输出必须包含：

1. 核心判断。
2. 支撑理由。
3. 最容易误判的点。
4. 赛中需要验证的信号。
5. 依据来源：赛前报告、`crawlerStatistics.js` 输出、本地数据、网络搜索或用户提供信息。
6. 专家最终结论。

每位专家的最终结论必须包含：

1. 赢盘方向：主队赢盘、客队赢盘或走盘，并写清对应盘口。
2. 大小球方向：大球、小球或谨慎观望，并写清对应大小球盘口。
3. 上半场走势：谁主动、是否容易有球、半场可能比分。
4. 下半场走势：谁会变招、谁会收缩、是否容易后段出球。
5. 全场比分预测：必须给出 1 到 2 个比分。
6. 结论理由：必须用该专家自己的视角说明为什么。

解说员和名宿必须形成观点张力。解说员负责大众叙事，名宿负责反大众风险。冲突不是问题，冲突本身要进入最终剧本。

### 第五步：生成 3 到 4 种比赛剧情

默认生成 4 种剧情：

1. 正路强势剧本：强队实力、盘口和场面一致。
2. 强队虚热剧本：强队控球多、名气大，但真实威胁不足。
3. 弱队偷袭剧本：弱队有清晰反击、定位球或消耗后偷袭路线。
4. 僵局消耗剧本：双方都不愿意过早冒险，比赛低风险运行。

如果比赛是强强对话、淘汰赛、杯赛末轮、必须分胜负等特殊场景，可以重命名剧情，但仍需保留 3 到 4 条不同走势。

每条剧情必须写清：

1. 成立条件。
2. 开局验证点。
3. 盘口验证点。
4. 可考虑的滚球方向。
5. 必须放弃该剧情的信号。

### 第六步：设计赛中观察时间轴

必须按时间段输出观察清单：

1. 0 到 10 分钟：看开局态度和真实压迫。
2. 10 到 25 分钟：看进攻质量和反击威胁。
3. 25 到 45 分钟：看剧情稳定性和半场前窗口。
4. 45 到 65 分钟：看教练调整和节奏变化。
5. 65 分钟以后：看体能、空间、情绪和赔率价值。

每个时间段都要说明：

1. 观察什么。
2. 哪些信号支持哪条剧情。
3. 哪些信号禁止下注。

### 第七步：给出入场纪律

任何滚球建议必须包含纪律边界：

1. 单场默认 500。
2. 真的很稳很稳，最多 1000。
3. 剧情不匹配，不下注。
4. 盘口失去价值，不下注。
5. 情绪不稳、疲劳、刚赢、刚输、刚扳平，不下注。

不允许因为「感觉很稳」直接建议重注。

### 第八步：保存赛前推演产物

生成的导演剧本必须保存为 Markdown 文件，和对应比赛的赛前分析报告放在同一个目录。

命名规则：

```text
{主队}_vs_{客队}_赛前推演.md
```

示例：

```text
cup-analyzer/theWorldCup/report/group-stage/round-1/突尼斯_vs_日本_赛前推演.md
cup-analyzer/epl/report/25-26/round-1/利物浦_vs_伯恩茅斯_赛前推演.md
```

保存规则：

1. 如果原赛前报告是 `{主队}_vs_{客队}.md`，则产物保存为同目录下的 `{主队}_vs_{客队}_赛前推演.md`。
2. 如果原赛前报告旁边有 `{主队}_vs_{客队}_cycle.html`，不要覆盖该文件，只在同级目录新增或更新 `_赛前推演.md`。
3. 如果同名 `_赛前推演.md` 已存在，先读取旧文件；除非用户明确要求重写，否则基于新分析更新内容，不删除用户已补充的赛中记录。
4. 最终回复必须告诉用户保存路径。

## 写作规则

1. 结论先行，再给理由。
2. 不编造不存在的数据、盘口或阵容。
3. 不把历史交锋、近期状态、未来赛程当作唯一主依据。
4. 不输出确定性语言，例如「必胜」「稳胆」「一定进球」。
5. 用「符合哪条剧情」表达判断，而不是只说「买哪边」。
6. 如果没有清晰入场点，最终建议必须是等待。
7. 使用网络搜索时标注来源，无法确认的信息不能写死。
8. 涉及代码逻辑时写必要注释，禁止嵌套三元表达式。
